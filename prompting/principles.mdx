## 提问三原则

好的提问需要满足三个条件。记住这三条，你和 Claude 的协作效率会大幅提升。

### 原则一：目标明确

告诉 Claude **具体要做什么**，而不是笼统地说"看看"。

| 不好的提问 | 好的提问 |
|-----------|---------|
| "帮我改一下这份文档" | "帮我把这份文档的第 3 节改成面向客户的语气，去掉所有技术术语" |
| "看看这个文件" | "读一下这个文件，列出里面所有提到的截止日期" |

### 原则二：约束清楚

告诉 Claude **不要做什么**或**范围限制**。

| 不好的提问 | 好的提问 |
|-----------|---------|
| "帮我写个总结" | "帮我写一个 200 字以内的总结，只包含业务影响，不要写技术细节" |
| "整理一下数据" | "整理这份数据，按月份分组，只看销售额大于 1 万的记录" |

### 原则三：格式要求

告诉 Claude **输出什么样的格式**。

| 不好的提问 | 好的提问 |
|-----------|---------|
| "帮我整理一下" | "帮我整理成表格形式，列包括：名称、状态、负责人、截止日期" |
| "分析一下" | "分析结果用编号列表呈现，每条不超过 30 字" |

<Tip>
  **记忆口诀**：**做什么 + 什么约束 + 什么格式** = 好的提问
</Tip>

## 跑偏了怎么办

Claude 可能会理解错你的意图，或者越改越偏。不要慌，有三种处理方法：

### 方法一：直接纠正（适合小偏差）

在对话里直接说明哪里理解错了：

```
不对，我的意思不是重写整段，而是只改最后一句的措辞。请重新来。
```

```
你理解错了，"客户"在这里指的是内部客户（其他部门），不是外部客户。
```

### 方法二：开新对话（适合大偏差）

如果对话已经乱了，最好开一个全新的对话重新开始：

- **Mac**：`Cmd + N`
- **Windows**：`Ctrl + N`

<Info>
  开新对话不会影响你的文件。你的文件内容不会因为对话重置而丢失。
</Info>

### 方法三：拒绝修改（适合文件被改错）

如果 Claude 修改了文件但改错了，在 Diff 视图里点 **Reject**（拒绝）即可恢复原样。这就是为什么[查看修改建议](/claude-code/core-usage#用法四查看并接受拒绝修改建议)这个功能这么重要。

<Tip>
  **官方建议**：指令越具体、越短，Claude 遵循的一致性越高。避免在一条消息里塞太多不相关的要求。一次只做一件事。
</Tip>
