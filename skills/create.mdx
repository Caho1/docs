## Skill 的文件结构

一个 Skill 在技术上其实就是**一个文件夹**，里面最核心的是一份 `SKILL.md` 文件：

```
my-skill/                    ← 一个文件夹
├── SKILL.md                 ← 核心：用自然语言写的操作指南（必须有）
├── templates/               ← 可选：输出模板（比如周报模板）
│   └── weekly-report.md
├── references/              ← 可选：参考资料（比如术语表、评审标准）
│   └── review-checklist.md
└── scripts/                 ← 可选：辅助脚本（技术同学编写）
    └── validate.sh
```

| 文件/目录 | 是否必须 | 说明 | 谁来写 |
|-----------|---------|------|--------|
| `SKILL.md` | 必须 | 操作指南，用自然语言写 | 业务 + 技术协作 |
| `templates/` | 可选 | 输出模板 | 业务人员提供 |
| `references/` | 可选 | 参考资料（术语表、检查清单等） | 业务人员提供 |
| `scripts/` | 可选 | 自动化脚本 | 技术同学编写 |

## SKILL.md 怎么写

用自然语言写，不需要任何代码。就像你给新来的实习生写一份操作手册一样。

### 基本结构

```markdown
---
name: weekly-report
description: 生成部门周报。当用户说"帮我做周报"、"整理本周工作"时自动触发。
---

# 部门周报生成流程

## 步骤
1. 读取用户提供的本周工作记录
2. 按以下分类整理：
   - 本周完成的事项
   - 进行中的事项（标注进度百分比）
   - 遇到的问题和风险
   - 下周计划
3. 用 templates/weekly-report.md 中的模板格式输出
4. 在末尾附上"需要领导关注的事项"清单

## 规则
- 每个事项用一句话概括，不超过 30 字
- 风险项必须标注影响范围和建议处理方式
- 不要编造内容，如果信息不够就标注"待补充"
```

### 关键部分说明

<AccordionGroup>
  <Accordion title="元数据（--- 之间的部分）">
    ```yaml
    ---
    name: weekly-report
    description: 生成部门周报。当用户说"帮我做周报"时自动触发。
    ---
    ```

    - **name**：技能名称，只能用小写字母、数字和连字符，最多 64 个字符
    - **description**：描述这个 Skill 做什么、什么时候触发。Claude 会根据这段描述判断什么时候该使用这个 Skill

    description 写得好不好，直接决定了 Claude 能不能在正确的时机触发 Skill。
  </Accordion>

  <Accordion title="步骤（告诉 Claude 怎么做）">
    用自然语言、编号列表写清楚操作步骤。就像给新同事写的操作手册。

    要点：
    - 步骤要按顺序排列
    - 每一步足够具体，不要模糊
    - 如果引用了模板文件，说清文件路径
  </Accordion>

  <Accordion title="规则（告诉 Claude 什么能做、什么不能做）">
    把你的"潜规则"和"注意事项"写成明确的规则。

    要点：
    - 用"必须"和"不要"这样明确的词
    - 规则要可验证（"不超过 30 字" 比 "简洁" 更好）
  </Accordion>
</AccordionGroup>

<Warning>
  根据官方建议，SKILL.md 的核心指令**控制在 200 行以内**。详细的参考资料放在 `references/` 目录下，Claude 只在需要时才读取，不会浪费上下文空间。
</Warning>
